### 소프트웨어 개발 보안

```
기밀성 : 시스템 내의 정보와 자원은 인가된 사용자한테만 접근이 가능합니다.
가용성 : 인가받은 사용자는 언제라도 사용할 수 있습니다.
무결성 : 시스템 내의 정보는 오직 인가된 사용자만 수정할 수 있습니다.
```

### 소프트웨어 개발 보안 관련 기관

```

1. 행정안전부
- 소프트웨어 개발 보안 정책을 총괄한다.
- 소프트웨어 개발 보안 관련 법규, 지침, 제도를 정비한다.
- 소프트웨어 보안 약점을 진단하는 사람의 양성 및 관련 업무를 수행한다.

2. 한국인터넷진흥원
- 소프트웨어 개발 보안 정책 및 가이드를 개발한다.
- 소프트웨어 개발 보안에 대한 기술을 지원하고, 교육과정 및 자격제도를 운영한다.

3. 발주기관
- 소프트웨어 개발 보안의 계획을 수립한다.
- 소프트웨어 개발 보안 사업자 및 감리법인을 선정한다.
- 소프트웨어 개발 보안의 준수 여부를 점검한다.

4. 사업자
- 소프트웨어 개발 보안 관련 기술 수준 및 적용 계획을 명시한다.
- 소프트웨어 개발 보안 관련 인력을 대상으로 교육을 실시한다.
- 자체적으로 보안 약점을 진단하고 제거한다.
- 소프트웨어 보안 약점과 관련된 시정 요구사항을 이행한다 
- 소프트웨어 개발 보안 가이드를 참조하여 개발한다.

5. 감리법인
- 감리 계획을 수립하고 협의한다.
- 소프트웨어 보안 약점의 제거 여부 및 조치 결과를 확인한다.

```

### Secure SDLC 

```
보안상 안전한 소프트웨어를 개발하기 위해 SDLC 에 보안 강화를 위한 프로세스를 포함 한 것을 의미합니다. 
```

### 시큐어코딩

```
소프트웨어 구현 단계에서 발생할 수 있는 보안 취약점들을 최소화 하기 위해
보안 요소들을 고려하며 코드를 구현하는 것으로,
보안 취약점을 사전에 대응하여 안정성과 신뢰성을 확보하기 위해 사용하는 것. 
```


### 세션 통제 

```
세션 = 서버와 클라이언트의 연결

1. 불충분한 세션 관리 발생 원인
- 일정한 규칙을 가지고 있는 세션 ID의 발급할 때
- 지나치게 긴 세션 타임아웃일 경우

2. 잘못된 세션에 의한 정보 노출
원인 : 다중 스레드 환경에서 멤버 변수에 정보를 저장할 때 발생하는 보안 약점
해결 방안 : 전역 변수를 사용하기 보다는, 지역 변수를 사용하여 변수의 범위를 설정하여 방지할 수 있다.

* 세션 설계시 고려 사항
- 로그아웃 요청시 할당된 세션이 완전히 제거되도록 한다.
- 시스템의 모든 페이지에서 로그아웃이 가능하도록 UI 구성한다.
- 세션 타임아웃의 경우, 중요도가 높으면 2~5분 중요도가 낮으면 15~30분 으로 설정한다.
- 이전 세션이 종료되지 않으면 새 세션이 생성되지 못하도록 설계한다. 
```

### 에러처리

```
소프트웨어 실행중 발생할 수 있는 오료들을 사전에 정의하여, 오류로 인한 문제들을
예방하기 위한 보안 점검 항목들이다.

1. 오류 메세지를 통한 정보 노출
2. 요류 상황 대응 부재
3. 부적절한 예외처리

```

### 개인키 암호화 기법

```
동일한 키로 데이터를 암호화 하고 복호화 한다. 

- 종류
 1. 블록 암호화 방식 : SEED, DES, AES, ARIA
 2. 스트림 암호화 방식 : LFSR, RC4

장점 : 암호화 / 복호화 속도가 빠르다! 
       알고리즘이 단순하다.
       공개키 암호 기법 보다 파일의 크기가 상대적 작다.
단점 : 사용자의 증가에 따라 관리해야 하는 키의 개수가 증가한다.

```

### 공개키 암호화 기법

```
데이터를 암호화 할 때 사용하는 공개키는 데이터베이스 사용자에게 공개하고 
복호화 할 때의 비밀키는 관리자가 비밀리에 관리한다.

장점 : 키의 분배가 용이하다.
       관리해야할 키의 개수가 작다
단점 : 암호화 / 복호화 속도가 느리다.
       알고리즘이 복잡하다.
       개인키 암호화 기법보다 비교적 파일이 크다

### 해시

```
임의의 길이의 입력 데이터나 메세지를 고정된 길이의 값이나 키로 변환하는 것을 의미한다.
```
